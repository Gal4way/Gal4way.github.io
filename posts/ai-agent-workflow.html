我如何用 AI Agent 搭建了一套覆盖研发全链路的个人工作流

2025-05-11 · Xu Yang · AI Agent Workflow Productivity

不用 AI 会被时代淘汰，沉迷 AI 会让自己退化。真正的高手是像优秀的 leader 一样驱动 AI，既借力提效，又保持自省和独立思考，最终变得又省力又强大。

系统全景

📡 热点推送        定时任务 → 全网检索 → 飞书文档（每日自动）
📖 论文审稿        PDF解析 → Review + Analysis + Read（三文档）
💻 代码开发        方案对齐 → 编码 → 强制自检 → commit
🖥️ GPU 实验调度   一键起机 → 自动运行 → 日志追踪
📋 实验文档管理    自动登记表格 → 创建文档 → 上传结果视频
📝 工作记录        每日自动总结 → 飞书文档 → 周报自动生成
🧠 跨会话记忆      Progress Log + Memory → 断点续做

下面按场景逐一展开。

一、热点推送：每日自动情报收集

通过定时任务（Cron Job）实现，每天早晨自动触发，无需人工干预。

[每日定时触发]
    │
    ▼
[全网检索过去 24h 热点]
    │  覆盖：AI前沿/开源生态、科技大厂动态、财经宏观、全球大事
    │  筛选：过滤营销稿，每类别最多 3 条，总量 8-12 条
    │
    ▼
[整理为高信息密度 Markdown]
    │  每条包含：加粗关键词标题、关键数据点、影响洞察、信息来源
    │
    ▼
[创建飞书文档] → 标题：YYYY_MM_DD_每日推送_热点

同时还有一个学术推送定时任务，专门抓取 AR Video Generation 领域的最新论文，筛选 3-5 篇高价值工作。

效果：每天早上打开飞书就能看到两份精选推送，不再需要手动刷 Twitter/ArXiv/新闻站。

二、论文阅读：三文档深度解读

封装为本地 Skill，输入一个 ArXiv 链接，自动产出三份飞书文档：

  • Review（英文）：模拟顶会审稿人视角，含 Strengths/Weaknesses/Rating
  • Analysis（中文）：技术深度分析，含评分理由和快速决策参考表
  • Read（中文）：Q&A 形式论文解读，8 个核心问题逐一拆解

Read 文档核心问题

#问题目的
Q1解决什么任务？快速定位论文 scope
Q2Technical challenge 有哪些？理解问题难点
Q3主要贡献点？判断创新度
Q4方法包含哪些模块？技术细节拆解
Q5模块如何解决 challenge？建立映射
Q6实验如何设计？评估严谨性
Q7Limitation？识别风险
Q8Future work？启发后续研究

三、代码开发：方案对齐 → 编码 → 强制自检

通过 CLAUDE.md（全局指令文件）+ PreToolUse Hooks 实现规范化开发流程。

[需求/想法]
    ▼
[/tech-doc 创建技术方案文档]  ← 飞书文档，含背景/方案/风险/ToDo
    ▼
[方案对齐确认]  ← 用户审阅文档，确认方向
    ▼
[编码实现]
    ├── 自动确认当前分支（必须在 yx/ 前缀分支上）
    ├── 最小改动原则，保持现有代码风格
    └── 新增代码必须有注释说明意图
    ▼
[强制自检]
    1. 逻辑正确性（条件分支、边界、off-by-one）
    2. 一致性（接口签名、参数透传链路）
    3. 副作用（是否影响未修改的功能路径）
    4. 遗漏（白名单、config、类型定义是否同步）
    ▼
[自我 Review]  ← 问自己：为什么这样做？为什么不那样做？
    （对保持技术 taste 和敏锐非常重要）
    ▼
[commit]  ← 格式：<type>: <description>

安全保障（三层防护）

层级机制作用
L1 Deny Listsettings.json 硬拦截禁止 git push/rm -rf/sudo 等
L2 Bash Hook运行时命令检查共享存储只读、merge 分支约束
L3 Write Hook文件级写保护非 yx/ 分支禁止修改 git 跟踪文件

四、GPU 实验调度：一键起机 → 自动运行

封装为 4 个 Slash Command Skill（/worker、/worker-status、/worker-login、/worker-run）。

[/worker 8]  ← 一键启动 8 卡 H100 worker
    ▼
[/worker-run train.sh]  ← 在 worker 上运行脚本
    │  自动生成 wrapper 脚本：
    │    1. 停止 GPU 占位进程
    │    2. cd 到仓库目录
    │    3. bash script.sh 2>&1 | tee 日志文件
    │    4. 恢复 GPU 占位进程
    ▼
[日志追踪]  ← /log/<仓库名>/MMDD_HHmm_脚本名.log
    ▼
[实验完成] → 触发实验文档管理流程

五、实验文档管理：自动登记 → 创建文档 → 上传结果

封装为 /experiment-record Skill，与飞书表格和文档深度集成。表格按 task 分组，千位前缀编号（0xxx=任务A、1xxx=任务B...），同组 ablation 实验紧邻排列。

实验表格结构

日期状态机器是否分析实验名称Setting 说明分析文档链接
5/9完成8×H100是1001_baselinerecent=100, oldest=19基准性能[链接]
5/9完成8×H100待做1002_recent3recent=3, oldest=19[链接]

工作流

[/experiment-record register]
    ├── 从推理脚本提取：OUTPUT_DIR、CKPT_PATH、关键超参
    ├── 从训练平台获取：机器配置、运行日期、状态
    ▼
[写入飞书表格]  ← 自动填充日期/状态/机器/实验名/setting
    ▼
[自动创建飞书文档]  ← 按模板：setting + 结果 + 结论
    ▼
[文档链接回填表格]
    ▼
[/experiment-record upload 1001]
    │  从 OUTPUT_DIR 找到 *.mp4，每个 setting 随机选 2 个视频
    │  上传到飞书文档对应章节
    ▼
[用户分析结论] → AI 填入表格和文档结论章节

实验文档模板

### Setting
- 目的：验证 context 压缩对推理效率的影响
- 关键参数：VIDEO_CTX_KEEP_LATENTS=3, COMPRESS_RATIO=4
- CKPT: hdfs://path/to/checkpoint

### 结果
[视频1] [视频2]  ← 自动上传，同组并排展示

### 结论
（由用户口述，AI 格式化填入）

六、工作记录：每日自动 → 周报自动

封装为 /work-log 和 /record Skill，与飞书文档集成。

每日工作记录

[完成一个子任务后，AI 主动提醒]
    │  "要记录到工作日志吗？"
    ▼
[AI 自动从对话上下文提取]
    │  时间、任务简述、内容、产出（commit/文档链接/实验编号）
    ▼
[展示给用户确认]
    ▼
[追加写入当天飞书文档]

飞书文档按周分文件夹，每天一个文档：

工作日志/
├── 26_5_5-26_5_11/     ← 按周分文件夹
│   ├── 26_5_5          ← 每天一个文档
│   ├── 26_5_6
│   └── ...
└── 26_5_12-26_5_18/

每条记录格式：

## [14:32] cherry-pick 压缩模块到 master

- **任务**: [rebase回master](飞书链接)
- **内容**: 完成压缩模块的 cherry pick，验证结果正常
- **产出**: commit `abc1234` on branch yx/rebase_to_master

周报自动生成

通过 /record weekly 自动汇总本周所有 progress 文件、实验表格、每日记录，生成结构化周报（本周完成的工作 + 实验进展和结论 + 下周计划），用户确认后写入飞书。

七、跨会话记忆：断点续做

AI Agent 的对话是无状态的——关闭窗口后，下次对话它什么都不记得。但研发任务往往跨越多天。解决方案是双层持久化：

层级存储内容生命周期
Memory~/.claude/memory/用户偏好、行为反馈、项目知识、易犯错误长期稳定
Progress~/.claude/progress/操作日志、当前状态、下一步、关联文档链接任务周期

Progress Log 示例

# Efficiency 验证

- 状态：活跃
- 分支：yx/ar_train_tf_history_compress
- 关联文档：
  - tech-doc: https://feishu.cn/docx/xxx
  - 实验表格行: 1001-1004
  - merlin trial: https://ml.xxx.net/jobs/62e1034310977b2a

## 操作记录
### 2026-05-09
#### 做了什么
- 在 pipeline 中完成 efficiency 插桩
- 创建 4 个测试脚本，提交 4 个 trial

#### 当前状态
- 4 个 trial 运行中，等待完成

#### 下一步
- 收集 efficiency.json，汇总对比表格

跨会话接力流程

[用户："继续做 efficiency 验证"]
    ▼
[AI 在 progress/ 下找到匹配文件]
    ▼
[读取关联的飞书技术文档] → 获取方案上下文
    ▼
[查询训练平台 trial 状态] → 获取最新运行状态
    ▼
[从"下一步"继续工作]  ← 零冷启动

效果：新对话不再需要花 5-10 分钟解释背景，说一句"继续"就能无缝接续。
